## 三、核心模块设计

### 3.1 特征提取网络 (FEN)

三个并行的 backbone 分别处理 RGB、Depth、Infrared：

- **RGB backbone**：标准 CNN/Transformer backbone（参考 [2] YOLO 系列）
- **Depth backbone**：结构同 RGB backbone，输入单通道 16bit depth 图
- **IR backbone**：结构同 RGB backbone，输入三通道 8bit 红外图（三通道视觉一致，本质堆叠的单通道热辐射灰度图）

可选方案：
- 三路共享权重（参数量小，但模态差异大时效果受限）
- 三路独立权重（参数量大，但每个模态提取最优特征）
- 前几层独立 + 后几层共享（折中方案）

### 3.2 Intra-Hypergraph Enhancement（模态内）

每个模态各自超图建模：
- **超边（hyperedge）** 连接多个空间节点，捕获模态内多对多高阶依赖
- 参考 M2I2HA 的设计：低秩分解 + 稀疏化降低计算量
- 输出：增强后的模态内特征

### 3.3 模态质量评估模块

参考 [3] EvaNet 的核心思想：

| 输入 | 处理 | 输出 |
|---|---|---|
| 每个模态的增强特征 | 轻量评估网络（质量打分） | 每个模态在当前场景下的质量分数 (0~1) |

评估依据：
- **RGB**：光照强度、对比度、模糊程度
- **Depth**：有效深度比例、深度一致性
- **Infrared**：热对比度、噪声水平

**评估模块在训练阶段的策略**：

| 策略 | 做法 | 适用场景 |
|---|---|---|
| 隐式学习 | 让网络自己学，通过损失函数中的质量权重回传 | 数据量大时 |
| 显式监督 | 人工标注模态质量分数做回归 | 需先验知识时 |
| 自监督 | 用模态间一致性做质量信号（如 RGB 和 IR 的边缘对齐度） | 无额外标注时 |

### 3.4 Inter-Hypergraph Fusion（跨模态融合）

三模态跨超图融合（扩展 M2I2HA 的双模态为三模态），基于质量分数动态调整：
- **质量分数高**：该模态特征在融合中占主导
- **质量分数低**：该模态特征权重被压低，减少噪声引入
- **质量分数极低**：可参考 [1] Channel Switching 的思路，直接屏蔽该模态

### 3.5 置信度计算

最终置信度由两部分决定：
```
confidence = alpha * cls_score + beta * fusion_quality_score

其中：
  cls_score      = 检测头输出的分类分数
  fusion_quality = 模态质量评估模块输出的融合质量综合评分
  alpha, beta    = 可学习的权重参数
```

这种设计的收益：
- 当融合质量高时，cls_score 和 quality_score 一致，置信度可靠
- 当融合质量低时，quality_score 自动压低置信度，避免"模态都崩了还输出假阳性"
- 满足题目要求："在一个或者多个模态数据成像质量较差时，算法也不应出现大幅性能波动"

## 七、论文 [5] 对比赛的启示与差别分析

论文 [5]（Zhu et al., NeurIPS 2025）是**组织者自己的 RGB-D-T 三模态工作**，是比赛最重要的技术参考。但其任务（跟踪）和你的任务（检测）有本质不同，需清楚地辨识哪些可以直接用、哪些需要改造。

### 7.1 硬件与数据层面（可直接复用）

| 条目 | [5] 的做法 | 比赛的对应 | 能否直接用 |
|---|---|---|---|
| 采集硬件 | ZED 双目（RGB+D）+ LGCS121 热成像 | ZED 双目 + InfiRay LG6122 热成像 | ✅ 几乎一致 |
| 模态对齐 | RGB 和 Depth 硬同步；Thermal 需特征点映射对齐 | **已对齐好**，直接发三张配对 PNG | ✅ 数据直接可用 |
| 图像格式 | 三模态 PNG，1920×1080 | PNG，未标注具体分辨率 | ✅ 大概率一致 |
| Depth 编码 | 单通道 16bit 无符号（毫米单位） | 单通道 16bit 无符号（毫米单位，0~19999 有效） | ✅ 完全一致 |
| TIR 编码 | 单通道复制成三通道输入 | 三通道 8bit 无符号，三通道视觉一致（本质单通道堆叠） | ✅ 一致 |

**结论**：数据层面的处理（读取、预处理、随机增强时保持三模态同步变换）可以直接复用 [5] 的 pipeline 思路。

### 7.2 训练策略层面（可借鉴但需改造）

**[5] 的做法**：
- 冻结 RGB-only 预训练 backbone（OSTrack），只训练 fusion + prompt 模块
- 仅 0.86M 可训练参数
- 60 epochs，每 epoch 60K 样本

**借鉴价值**：极高，但需适配到检测任务。

| [5] 的做法 | 你的改造方案 |
|---|---|
| 用 OSTrack（RGB 跟踪）做预训练 backbone | 用 YOLOv8/v11/v13，在 COCO/Objects365 上预训练的权重 |
| 冻结 backbone + 只训 fusion | 冻结 backbone+neck 的前几层，只训融合模块 + detection head |
| Prompt Learning 注入 RGB 分支 | 替换为 Inter-Hypergraph Fusion + 质量评估模块做主干融合 |
| Depth-TIR 正交投影约束 | 可加入 Inter-Hypergraph Fusion 中，作为跨超边生成的约束条件 |
| 仅 0.86M 可训练参数 | 限制只有新增的融合模块是可训练的——在 2000 张图上不这么做容易过拟合 |

**关键区别**：跟踪模型只在第 1 帧见过目标，后续帧只是定位，所以可以冻结 backbone。检测需要在每个推理中识别并定位多类多目标，detection backbone 也需要一定程度的 fine-tune。

### 7.3 任务层面的本质差别（不可直接照搬）

| 维度 | [5] RDTTrack（跟踪） | 你的比赛（检测） |
|---|---|---|
| 任务定义 | 给定第 1 帧目标位置，后续帧跟踪该目标 | 每帧独立检测所有 12 类目标 |
| 输出 | 每帧 1 个 bbox | 每帧多个 bbox（上限 100） |
| 输出格式 | [x, y, w, h] | [class_id, norm_center_x, norm_center_y, norm_w, norm_h, confidence] |
| 类别 | 跟踪开始才知道是哪个，不输出类别 | 需输出 12 类分类结果 |
| 指标 | AUC（成功率）+ DP（精度） | mAP@50-95（检测标准指标） |
| 模型架构 | OSTrack（Transformer 编码器 + 预测头） | YOLO 系列（backbone + neck + 检测头） |
| 推理方式 | 利用时序信息（帧间关联） | 单帧独立推理 |
| 关键挑战 | 目标外观变化、遮挡后重识别 | 密集小目标、类别不平衡、低光/遮挡下仍能识别 |

**直接照搬 [5] 的架构会出问题的点**：

1. **OSTrack 不适合检测**——它设计成每帧只输出一个 bbox，无法处理多目标
2. **Prompt Learning 的输入结构不匹配**——它假设 RGB 特征占主导，D+T 只是 prompt。检测中 D 和 T 有时是主要线索（如低光下 RGB 基本无用），不能当"辅助 prompt"处理
3. **没有分类能力**——RDTTrack 不输出类别，只定位。12 类分类是检测特有的

### 7.4 整体方案的定位

你的完整方案架构图：

```
├── 检测框架：YOLOv8/v11/v13（来自 M2I2HA）
├── 三模态特征提取：三路 backbone（共享或独立权重）
├── 模态内增强：Intra-Hypergraph Enhancement（来自 M2I2HA）
├── 模态质量评估：EvaNet 风格（来自 [3]）
├── 跨模态融合：Inter-Hypergraph Fusion（来自 M2I2HA）
│   └── 可选约束：正交投影（来自 [5] 的思想）
├── 置信度校调：质量分数 + 分类分数（方案创新点）
├── 训练策略：冻结 backbone 低层，冻结 COCO 预训练权重（来自 [5] 的思路）
└── 公开数据验证：DroneVehicle/FLIR/LLVIP（方案验证层）
```

**一句话**：你的方案 = M2I2HA 的检测架构骨架 + [5] 的训练策略和数据经验 + EvaNet 的质量评估思想。

## 八、Prompt Learning vs. 超图融合——本质差别

这是方案设计中最关键的一个抉择。两套思路的根本分歧在于：**多模态之间是主从关系还是对等关系。**

### 8.1 Prompt Learning 的隐含假设

Prompt Learning（来自 [5] RDTTrack / ViPT 系列）的结构本质上是在回答一个问题：**"我有一个非常强的 RGB 模型，如何在不动它的情况下把 D 和 T 的信息塞进去？"**

```
RGB → [预训练 backbone ← 冻结] ──→ 强特征
                                       ↑
D ──→ 正交投影 ──→ D-T 融合 ──→ Prompt 生成 ──→ 注入各层
T ──→                    ┘                       ↑
                                           RGB 是主厨
                                           D/T 只是调料包
```

**隐含假设**：
1. **RGB 是主模态**——模型的大部分能力来自 RGB，D 和 T 只是"锦上添花"
2. **D 和 T 的信息可以压缩成少量 prompt tokens**——注入到 Transformer 层就够了
3. **RGB backbone 不需要适配**——冻结的原始权重就能处理加了 prompt 的输入

**这在什么场景下成立**：跟踪任务。跟踪起手就知道目标是什么，主要靠 RGB 纹理做匹配，D 和 T 只是在 RGB 退化时拉一把的辅助信号。

### 8.2 超图融合的隐含假设

超图融合（来自 M2I2HA）的假设完全不同：**"这三种模态没有一个先验地主，需要根据场景动态决定谁说了算。"**

```
RGB ──→ backbone ──→ Intra-HG ──┐
                                  ├──→ Inter-HG（对等融合）──→ 检测头
D   ──→ backbone ──→ Intra-HG ──┤
                                  │    三个人坐下来开会
T   ──→ backbone ──→ Intra-HG ──┘    谁有理听谁的
```

**隐含假设**：
1. **模态对等**——决策时 RGB 不一定比 D 或 T 更有权威
2. **关系是多对多的**——一个区域在 RGB、D、T 三张图上的特征之间不是线性关系，需要用超边建模
3. **融合需要双向**——D 的信息影响 T 的解读，T 的信息也能修正 D 的误解

### 8.3 核心差异一列子

| 维度 | Prompt Learning | 超图融合 |
|---|---|---|
| **模态地位** | RGB 为主，D/T 为辅（主从关系） | RGB/D/T 对等（并列关系） |
| **信息流向** | 单向注入（D/T → RGB） | 双向交换（RGB ↔ D ↔ T） |
| **信息密度** | D+T 压缩成少量 prompt tokens | D 和 T 保留完整的空间特征图 |
| **关系建模能力** | 仅 pairwise（attention 模式） | 多对多高阶关系（超边模式） |
| **对 RGB 退化的表现** | RGB 主干是冻结的弱特征 → 差 | 质量评估压低 RGB 权重 → 仍好 |
| **模态缺失处理** | 需额外设计 missing modality 策略 | 超图天然支持部分节点缺失 |
| **计算开销** | 小（仅新增 prompt 参数） | 较大（三路 backbone + 超图层） |
| **参数量** | 约 0.86M 可训练参数 | 完整训练则大，冻结则可控 |
| **适用场景** | RGB 占主导，辅助模态锦上添花 | 各模态在不同场景下轮流主导 |

### 8.4 为什么比赛场景下超图融合更合适

题目原文中的两个关键约束条件直接宣告了 Prompt Learning 的假设不成立：

> *"在一个或者多个模态数据成像质量较差时，算法也不应出现大幅性能波动"*

这句话翻译成多模态关系就是：**哪个模态主导，取决于场景，而不是你的架构写死的。**

具体场景实测：

| 场景 | RGB | Depth | Thermal | 应该谁主导 | Prompt Learning 会怎样 |
|---|---|---|---|---|---|
| 白天、光照好 | ✅清晰 | ✅几何 | ✅温差 | RGB 为主 | ✅ 正常工作 |
| 夜间 | ❌暗 | ✅几何 | ✅高温物体突出 | **T 为主** | ⚠️ 冻结的 RGB backbone 输出弱特征，prompt 只是"调味料"压不住 |
| 烟雾/雨雾 | ❌模糊 | ✅几何 | ✅透过烟雾 | **D+T 为主** | ⚠️ 同上，污染了主干特征 |
| 热交叉（夏天人和环境等温） | ✅清晰 | ✅几何 | ❌温差低 | **RGB+D 为主** | ⚠️ T 的 prompt 引入了错误线索 |
| 玻璃/镜面 | ✅反射干扰 | ❌无效 | ✅温度正常 | **RGB+T 为主** | ✅ D 的 prompt 弱化了影响 |

Prompt Learning 在 RGB 完好的情况下没问题。但一旦出现**夜间、雨雾等 RGB 主导失效的场景**，冻结的 RGB backbone 本身就是瓶颈——它提取坏的 RGB 特征，prompt 信号只能做小幅度调整，拉不回来。

超图融合的路线不一样：质量评估模块感知到 RGB 质量低 → 自动压低 RGB 分支的融合权重 → D 和 T 的特征在 Inter-Hypergraph 中成为主导 → 检测结果不崩。

### 8.5 一句话

**Prompt Learning 假定 RGB 始终是可靠的会议召集人；超图融合假定任何人都能当会议召集人，取决于今天的议题。比赛规定你必须应付"任何一个人都可能今天不在状态"的场景，所以选超图融合。**

## 九、全流程问题清单

以下按数据流顺序列出每个环节当前还存在的问题或待决策项。🟡 = 可解决 ⚠️ = 需要认真设计 🔴 = 核心风险

### 9.1 数据输入与预处理

| # | 问题 | 优先级 | 说明 |
|---|---|---|---|
| 🟡 | **Depth 的无效区域怎么处理？** | 中 | Depth 值为 0 或过小代表无效区域（如远距离、镜面反射）。直接喂 0 会误导模型。需要 masking 或插值补全 |
| 🟡 | **Thermal 三通道堆叠是冗余的，要不要压缩？** | 低 | 论文里说"三通道视觉特征高度一致，无实际彩色语义，本质为 3 个单通道热辐射灰度图像堆叠"。可以喂 1 通道，或继续 3 通道拷贝（backbone 设计更方便） |
| 🟡 | **Depth 16bit 怎么归一化？** | 中 | RGB 和 Thermal 是 8bit [0,255]，Depth 是 16bit [0,19999]，数值范围差了两个数量级。直接 /19999 归一化到 [0,1] 会丢失近处物体的细节——相机在 0.3m~2m 范围内的 Depth 变化只占整个范围的很小一段 |
| ⚠️ | **12 类数据的分布怎么样？** | 高 | 还不知道。person 和 car 可能占大头，animal、ball、uav 可能是长尾。这直接影响损失函数设计和数据采样策略。拿到数据后第一件事就是看类别分布 |
| 🟡 | **数据增强时三模态同步问题** | 中 | Mosaic、MixUp、RandomAffine 等在 YOLO 中极其重要，但每个操作都必须三模态 + bbox 同时做完全相同的变换。Mosaic 尤其麻烦——4 张图拼成 1 张，每张的 3 个模态要各自拼好后再对齐 |

### 9.2 Backbone 初始化（🔴 核心风险）

| # | 问题 | 优先级 | 说明 |
|---|---|---|---|
| 🔴 | **Depth 和 Thermal 的 backbone 权重从哪来？** | **高** | RGB backbone 可以用 ImageNet/COCO/Objects365 预训练权重。但 Depth 和 Thermal——没有可用的标准预训练模型。2000 张图从头训练必定过拟合 |
| | 几个可选方案： | | |
| | 方案 A：用 RGB 预训练权重初始化 Depth/Thermal backbone | 最直接 | Depth 单通道 → 复制成 3 通道喂入；Thermal 本来就是 3 通道。ImageNet 特征（边缘、纹理）对 Depth 和 Thermal 也有一定迁移性 |
| | 方案 B：三路共享权重 | 参数量最小 | 但 RGB 的纹理模式 ≠ Depth 的几何模式 ≠ Thermal 的热模式，共享可能会互相拖累 |
| | 方案 C：各模态在各自公开数据上预训练 | 条件不允许 | 比赛禁止私自扩充外部数据，但没说不能用其他数据做预训练？需要确认 |
| | 方案 D：三路独立 + RGB 初始化其他两路 | 建议 | 用 ImageNet 权重初始化三个 backbone，然后让各 modality 自己 fine-tune 出差异。浅层（边缘检测）可能共享，深层（语义）各自适应 |
| 🟡 | **Backbone 冻结策略** | 中 | 参考 [5] 的思路：冻结低层（前几个 stage），只训练高层 + 融合模块。但检测和跟踪不同，detection head 需要充分训练。建议：冻结 backbone 的 stem+stage1-2，训练 stage3-4+neck+head+fusion |

### 9.3 Intra-Hypergraph Enhancement

| # | 问题 | 优先级 | 说明 |
|---|---|---|---|
| ⚠️ | **实现细节不明确** | 高 | M2I2HA 的代码未公开。论文里描述了低秩分解、稀疏化、FuseSEBlock、LR-C3AH Block、DSC3k Block，但这些都是嵌入在 YOLOv8/v13 架构里的，复现有一定工作量 |
| 🟡 | **超图大小怎么定？** | 中 | 超节点数 = 特征图的空间位置数（H×W）。超边数 m 和稀疏率 γ 需要调参。论文没有给出具体值 |
| 🟡 | **三个模态的超图超参数是否独立？** | 低 | 理论上每个模态可以有不同的超边数，但为了实现简洁可以先统一 |

### 9.4 模态质量评估模块（⚠️ 核心设计）

| # | 问题 | 优先级 | 说明 |
|---|---|---|---|
| ⚠️ | **没有质量标签，怎么训练？** | **高** | EvaNet 是监督训练（有融合图像和主观评分）。比赛没有模态质量的 ground truth。需要无监督/自监督方案 |
| | 候选方案： | | |
| | 方案 A：模态间一致性作为质量信号 | 推荐 | 如果 D 和 RGB 的边缘对齐度高 → Depth 质量好；如果 T 和 RGB 的温度梯度一致 → Thermal 质量好。不需要人工标注 |
| | 方案 B：利用检测损失隐式学习 | 可行 | 把质量分数作为一个可学习的参数，让检测损失自动调节。质量低的模态在融合中权重自然下降 |
| | 方案 C：人工规则 | 先用着 | RGB：计算拉普拉斯方差（模糊度）；Depth：统计有效像素比例；TIR：计算像素值方差（热对比度）。规则式不做为最终方案，但 baseline 够用 |
| 🟡 | **质量评估模块的架构设计** | 中 | 轻量级——每个模态的特征图过一个小型 CNN + 池化 + FC，输出一个标量 (0,1)。不能太大（避免增加过多参数量） |

### 9.5 Inter-Hypergraph Fusion（三模态扩展）

| # | 问题 | 优先级 | 说明 |
|---|---|---|---|
| ⚠️ | **双模态 → 三模态的超图扩展** | **高** | M2I2HA 只做了两个模态间的 Inter-Hypergraph Fusion。三模态需要新的跨超边结构：一条超边同时连接 RGB、D、T 三组节点中的多个点。公式需要重新推导，但超图理论本身是定义好的，数学上可行 |
| 🟡 | **质量分数如何融入超图？** | 中 | 方案：质量分数作为超边权重的 bias 项——质量高的模态的超边初始权重高，融合时影响更大 |

### 9.6 置信度校调

| # | 问题 | 优先级 | 说明 |
|---|---|---|---|
| 🟡 | **confidence = f(cls_score, quality_score) 的具体形式？** | 中 | 加权和：α×cls + β×quality？乘积：cls × quality？α、β 固定还是可学习？建议从加权和开始，α、β 可学习 |
| 🟡 | **置信度校调的效果怎么验证？** | 中 | 看 mAP@50-95（它本身就依赖 confidence 排序）。如果质量评估模块有效，加了 quality_score 后的 mAP 应该高于只用 cls_score 的版本 |

### 9.7 训练策略

| # | 问题 | 优先级 | 说明 |
|---|---|---|---|
| ⚠️ | **2000 张图，12 类——过拟合风险极高** | **高** | 需要：强数据增强 + 冻结策略 + weight decay + dropout/layer norm + early stopping |
| 🟡 | **验证集划分** | 中 | 2000 张训练集，建议 8:2 或 9:1 分训练/验证。不能动测试集（比赛规定） |
| 🟡 | **损失函数设计** | 中 | 基础：分类损失（Focal Loss）+ 回归损失（CIoU/GIoU）。额外的：质量评估模块的损失（自监督一致性损失） |
| 🟡 | **训练效率** | 低 | 三路 backbone 并行计算量大。如果输入是 640×640 三路，单卡可能需要 8-16 的 batch size |
